# Wonder illustrations are AI-generated, human-vetted, and committed as assets

## Status

superseded by ADR-0007 — Wonders now use real openly-licensed **photos**
(Wikimedia Commons), not AI illustrations. Note: the illustrations described
below were never actually rendered/committed (only the generator + a prompt
manifest existed; the live fallback was always the emoji), so nothing shipped was
discarded. The build-time + contact-sheet + emoji-fallback machinery here is
reused by ADR-0007.

## Context & decision

Issue #23 asks that every **Wonder** carry a picture, not just an emoji. ADR-0002
governs **text** content: hand-authored drafts run through a deterministic,
rule-based **Content Guardian** and regenerate `src/data` reproducibly. Images
do not fit that model — the rule Guardian cannot judge a picture's content, and a
strong illustration cannot be produced by hand at the scale of ~144 wonders.

We decided to add wonder pictures as a **new, separate content type**:

- A consistent **children's-atlas illustration** per wonder (one shared look,
  per the brand in PRODUCT.md — refined, jewel-toned, never photo-realistic
  clutter), generated by `scripts/generateWonders.mts`. That script uses the AI
  SDK `experimental_generateImage` via **Vercel AI Gateway** (`AI_GATEWAY_API_KEY`,
  model `openai/gpt-image-1`), so the provider is swappable by a string.
- Output is **PNGs committed to the repo** at `public/wonders/<code>-<slot>.png`
  (`slot` ∈ `landmark | animal | food`). The committed assets — not a regenerable
  pipeline — are the source of truth, so the app stays static and offline.
- The app loads images **by this filename convention**, not via a stored field —
  so the Guardian text pipeline and `countryContent.json` stay untouched, and
  dropping a PNG into `public/wonders` is all it takes for a picture to appear.
  When a file is absent the Wonders station **falls back to the emoji** (Design
  Principle 3), so coverage and pictures land independently and a missing image
  never breaks a journey.
- Vetting is a **one-time human glance** at the `/wonders-sheet` contact sheet, in
  the spirit of ADR-0002's "the family glances at the one item the machine is
  unsure of," scaled to the image batch. The rule Guardian is **not** extended to
  pixels.
- Auth uses a single `AI_GATEWAY_API_KEY` on the existing Vercel account (no new
  third-party signup); the generator is `npm run gen:wonders` and supports
  `--dry-run` (prompt manifest only) and `--only=CODE` (pilot a subset).

## Consequences

- This is the first content that is **not** reproducible by `npm run author:content`.
  ADR-0002's "re-running regenerates `src/data`" holds for text only; images
  regenerate via their own script and are then committed.
- A new build-time dependency on an image API + `AI_GATEWAY_API_KEY` (author-time
  only; nothing about it ships in the live bundle).
- Image safety rests on the style-locked prompt plus the human contact-sheet pass,
  not on the automated Guardian.
